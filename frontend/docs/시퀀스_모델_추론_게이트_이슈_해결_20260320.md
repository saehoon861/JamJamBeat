# 시퀀스 모델 추론 게이트 이슈 해결 기록 (2026-03-20)

## 1) 문제 요약

시퀀스 모델을 기본으로 켠 상태에서 MediaPipe 초기화는 성공했지만, 실제 런타임에서 손 추론이 끊기거나 매우 낮은 빈도로만 동작하는 문제가 있었다.

관측 로그 핵심:

- `Graph successfully started running.`
- `[MediaPipe] init:createFromOptions primary success`
- `[Perf][HandTracking] frames: ... inferences: 0`

즉, **MediaPipe 자체 초기화 실패가 아니라, 추론 루프 게이트 조건 문제**였다.

---

## 2) 원인 분석

### A. 시퀀스 모델 함수 주입 누락

`resolveGesture`가 고정 import된 모델 함수 경로를 참조하던 구조라, 동적으로 선택된 시퀀스 모델 함수가 제스처 판정 경로에 안정적으로 연결되지 않았다.

관련 파일:

- `frontend/src/js/gestures.js`
- `frontend/src/js/main.js`

### B. no-hand 시 상태 정리 누락

손이 사라졌을 때 제스처 안정화 상태와 시퀀스 버퍼가 완전히 정리되지 않아, 복귀 시점에 이전 상태가 남을 수 있었다.

관련 파일:

- `frontend/src/js/interaction_runtime.js`
- `frontend/src/js/model_inference_sequence.js`

### C. 추론 게이트가 `video.currentTime` 신호에 과도 의존

일부 실행 환경에서 live stream의 `currentTime` 신호가 불안정/정체되며, `hasFreshFrame` 조건이 거의 성립하지 않아 추론이 멈추는 케이스가 발생했다.

관련 파일:

- `frontend/src/js/hand_tracking_runtime.js`

### D. WebGL 경고 해석 혼선

`OpenGL error checking is disabled`는 단독으로는 실패 신호가 아니라 정보성 로그다.

실패 여부는 다음과 같이 구분해야 한다.

- 성공 신호: `Graph successfully started running`, `init:... success`
- 실패 신호: `StartGraph failed`, `CalculatorGraph::Run() failed`, WebGL context lost

## 3) 수정 내용

### 3-1. 시퀀스 모델 함수 주입 구조로 변경

- `frontend/src/js/gestures.js`
  - `modelPredictionProvider` 도입
  - `setModelPredictionProvider()` 추가
  - `resolveGesture()`가 provider를 통해 모델 예측 함수 호출
  - `resetGestureState()` 추가
- `frontend/src/js/main.js`
  - 동적 모델 import 이후 `setModelPredictionProvider(getModelPrediction)` 호출

효과:

- 시퀀스 모델이 실제 제스처 판정 경로에 일관되게 연결됨

### 3-2. no-hand 시 상태 리셋 강화

- `frontend/src/js/interaction_runtime.js`
  - 추적 리셋 시 `resetGestureState(handKey)` 호출
  - `getModelPrediction(null, resetAt, handKey)` 호출로 no-hand 경로 강제 반영

효과:

- 손 재진입 시 이전 버퍼/안정화 상태가 섞일 가능성 감소

### 3-3. 추론 게이트 복구 로직 보강

- `frontend/src/js/hand_tracking_runtime.js`
  - `FORCE_INFERENCE_IDLE_MS` 유지 (완전 정지 방지)
  - `FRAME_SIGNAL_STALE_MS` 추가
  - `requestVideoFrameCallback` 기반 frame signal probe 추가
  - `shouldRunInference()`를 사유 기반 게이트로 재구성
    - `fresh_frame`
    - `time_fallback`
    - `idle_recovery`
  - perf 카운터 확장
    - `forcedInferences`
    - `timeFallbackInferences`
  - slow frame 경고 임계값을 쿼리 파라미터로 조절 가능하게 변경 (`slowFrameWarnMs`, 기본 40ms)

효과:

- `currentTime` 신호 불안정 환경에서도 추론이 0으로 고정되지 않음
- fallback이 왜 발동했는지 로그로 분리 가능

## 5) 운영 시 확인 포인트

런타임 콘솔에서 다음을 같이 본다.

1. 초기화 성공 여부

- `Graph successfully started running`
- `[MediaPipe] init:createFromOptions ... success`

2. 추론 진행 여부

- `[Perf][HandTracking]`의 `inferences`가 지속적으로 증가하는지

3. 게이트 상태 해석

- `forcedInferences`가 계속 높은 경우: 프레임 신호가 장시간 정체되어 idle 복구 경로 의존
- `timeFallbackInferences`가 높은 경우: frame signal stale 감지 후 시간 기반 fallback 경로 작동

## 6) 튜닝 파라미터

문제 재현/완화 테스트용 쿼리 파라미터:

- `inferFps`
- `forceInferIdleMs`
- `frameSignalStaleMs`
- `slowFrameWarnMs`

권장 원칙:

- 증상 재현 시 먼저 `inferences`, `forcedInferences`, `timeFallbackInferences`의 비율을 보고 병목을 판단한다.
