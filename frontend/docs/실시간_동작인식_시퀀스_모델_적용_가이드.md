## 1) 목적

실시간성을 유지하면서 안정적으로 적용하는 방법을 정리합니다.

핵심 목표
- 추론 지연을 과도하게 늘리지 않기
- 오인식 흔들림을 줄이면서 반응성을 유지하기

## 2) 빠른 결론 (MVP 권장안)
가장 안전한 시작점은 슬라이딩 윈도우 + LSTM 또는 GRU + confidence gating 조합입니다.

권장 기본값
- inferFps: 18
- window(T): 16 프레임
- hop: 2 프레임 (약 9Hz 예측)
- EMA alpha: 0.4
- enterThr: 0.75
- exitThr: 0.55
- cooldownMs: 250

## 4) 실시간 추론 파이프라인 (권장 구조)
1. MediaPipe에서 프레임별 랜드마크 추출
2. 프레임 feature 벡터(F) 생성
3. 손 단위 버퍼(Map<handKey, ring buffer>)에 push
4. 버퍼 길이가 T에 도달하면 [1, T, F] 텐서 구성
5. ONNX Runtime Web로 시퀀스 추론
6. EMA/히스테리시스/쿨다운으로 이벤트 확정

중요 포인트
- 학습/추론의 feature 순서, 정규화, 패딩 규칙은 반드시 동일해야 합니다.
- 다손/손 미검출 상황에서도 입력 크기는 고정되어야 합니다(0 패딩).

## 5) 브라우저 ONNX Runtime Web 예시 코드
~~~js
import * as ort from "onnxruntime-web";

const T = 16;
const HOP = 2;
const perHandBuffer = new Map();
let hopCounter = 0;

function pushFrameAndMaybePredict(handKey, frameFeature, session, inputName) {
  const buf = perHandBuffer.get(handKey) ?? [];
  buf.push(frameFeature);
  if (buf.length > T) buf.shift();
  perHandBuffer.set(handKey, buf);

  if (buf.length < T) return null;
  if ((hopCounter++ % HOP) !== 0) return null;

  const F = frameFeature.length;
  const flat = new Float32Array(T * F);
  for (let t = 0; t < T; t++) flat.set(buf[t], t * F);

  const x = new ort.Tensor("float32", flat, [1, T, F]);
  return session.run({ [inputName]: x });
}
~~~

sequence = []

## 8) 지연(레이턴시) 관점 체크
결정 지연은 대략 아래로 생각할 수 있습니다.

decision_delay ~= (T / inferFps) + smoothing_delay + cooldown

예시(T=16, inferFps=18)에서 윈도우 자체 지연은 약 0.89초입니다.
반응성이 부족하면 다음 순서로 조정합니다.
1. T 감소
2. hop 감소
3. velocity/차분 feature 추가(짧은 T 보완)

## 9) 흔한 실패 원인
- 학습 feature 스키마와 추론 feature 스키마 불일치
- 손 좌/우 순서가 프레임마다 바뀌는 문제
- 손 미검출 프레임에서 텐서 shape 변형
- threshold만 높이고 히스테리시스/쿨다운이 없는 경우

## 10) 업그레이드 로드맵
1. MVP: LSTM/GRU + sliding window + gating
2. 경량화: 모델 축소/양자화(INT8) 검토
3. 고급화: CoST-GCN 등 continual/online skeleton 모델로 전환

MVP 단계에서 API를 pushFrame(features) -> maybeEmitGesture() 형태로 고정해두면,
모델 교체 시 프론트 로직 변경을 최소화할 수 있습니다.
