# 학습 모델 웹서버 배포를 위한 정리 문서

## 1. 모델 입력 형식

### 1-1. 기본 입력 shape
이 모델의 기본 입력 shape은 다음과 같다.

```text
(1, 21, 3)
```

의미는 다음과 같다.

- `1`: 배치 크기
- `21`: 손 랜드마크 개수
- `3`: 각 랜드마크의 좌표 차원 `(x, y, z)`

즉, 모델은 **손 랜드마크 21개의 3차원 좌표**를 한 프레임 단위로 입력받는다.

---

### 1-2. 입력 데이터 타입
입력 타입은 `float32`이다.

웹에서 ONNX Runtime Web을 사용할 경우 일반적으로 `Float32Array` 형태로 입력을 만들어야 한다.

---

### 1-3. feature 개수
한 프레임은 본질적으로 `21 × 3 = 63`개의 숫자로 구성된다.

즉 한 프레임의 원래 표현은 다음과 같이 볼 수 있다.

```text
63차원 벡터
```

이를 모델 입력 시 다음과 같이 reshape하여 사용한다.

```text
(63,) -> (21, 3)
```

따라서 브라우저에서 MediaPipe 결과를 모델에 넣을 때도, 본질적으로는 다음 정보가 필요하다.

- 랜드마크 21개
- 각 랜드마크의 `x, y, z`
- 총 63개 값
- **학습 때와 동일한 순서**

---

## 2. 모델이 사용하는 입력값의 의미

이 모델은 이미지 자체를 입력받는 것이 아니라, **랜드마크 좌표 숫자만 입력으로 받는 모델**이다.

입력의 의미는 다음과 같다.

```python
[
  [x0, y0, z0],
  [x1, y1, z1],
  ...
  [x20, y20, z20]
]
```

즉, 한 손에 대한 21개 랜드마크 좌표 배열이 입력된다.

---

## 3. 이 모델은 시퀀스 모델이 아니라 프레임 단위 모델

정리하면 이 모델은 **시퀀스(sequence) 모델이 아니라 단일 프레임 추론용 모델**이다.

### 핵심 내용
- 프레임 여러 개를 묶어서 넣는 구조가 아니다.
- 현재 구조에서는 시퀀스를 만들지 않는다.
- 웹에서도 영상 전체를 묶어 넣는 것이 아니라, **프레임마다 한 번씩 추론하는 방식**으로 이해해야 한다.

즉:

- 입력 단위: 프레임 1개
- 출력 단위: 프레임 1개에 대한 클래스 점수

---

## 4. 라벨 정보

라벨 컬럼 이름은 다음과 같다.

```python
y = df["gesture"].to_numpy(dtype=np.int64)
```

즉 정답 클래스는 `gesture` 컬럼에 저장되어 있다.

---

## 5. 웹 추론을 위해 반드시 알아야 하는 입력 제약

### 5-1. 랜드마크 개수 고정
모델은 랜드마크 개수가 반드시 21개라고 가정한다.

```python
assert N == self.num_landmarks
```

즉, 웹에서도 반드시 21개 랜드마크를 넣어야 한다.

---

### 5-2. 좌표 차원 고정
모델은 각 랜드마크가 반드시 3차원 좌표라고 가정한다.

```python
assert C == self.coord_dim
```

즉 `(x, y, z)` 3개 값이 필요하다.

---

### 5-3. 랜드마크 순서가 학습 때와 반드시 같아야 함
모델은 랜드마크 인덱스별 의미를 학습한다.

```python
landmark_ids = torch.arange(self.num_landmarks, device=x.device)
landmark_tokens = self.landmark_embed(landmark_ids)
x = x + landmark_tokens
```

이 말은 곧 다음을 의미한다.

- 0번 랜드마크는 항상 wrist
- 1번 랜드마크는 항상 thumb_cmc
- ...
- 20번 랜드마크는 항상 pinky_tip

즉 **웹에서 랜드마크 순서가 바뀌면 모델은 완전히 다른 의미로 받아들인다.**  
shape만 맞춘다고 되는 것이 아니라, **순서까지 동일해야 한다.**

---

## 6. 출력 형식

모델 출력은 다음과 같다.

```python
logits = self.classifier(feat)  # (B, num_classes)
return logits
```

기본 설정에서 `num_classes = 7`이므로, 배치 크기 1일 때 출력 shape은 다음과 같다.

```text
(1, 7)
```

---

### 출력값의 의미
출력은 확률(probability)이 아니라 **logits**이다.

즉 웹에서 받는 값은 다음 중 후자가 맞다.

- 확률값 ❌
- 정규화되지 않은 클래스 점수(logits) ✅

따라서 웹에서 최종 클래스를 사용하려면 일반적으로 다음 과정을 거친다.

- argmax로 최대 점수 클래스 선택
- 또는 softmax를 적용해 확률로 변환 후 사용

---

## 7. 모델 내부 구조 요약

### 7-1. 좌표 임베딩
좌표 `(x, y, z)`는 바로 Transformer에 들어가는 것이 아니라 먼저 임베딩된다.

```python
self.coord_embed = nn.Sequential(
    nn.Linear(coord_dim, d_model),
    nn.ReLU(),
    nn.Linear(d_model, d_model),
)
```

즉 각 랜드마크의 3차원 좌표를 `d_model` 차원으로 변환한다.

예를 들어 현재 기본값이 `d_model=64`라면:

- 입력: 3차원
- 출력: 64차원 임베딩

---

### 7-2. Transformer Encoder
그 다음 Transformer Encoder가 랜드마크들 사이의 관계를 학습한다.

```python
encoder_layer = nn.TransformerEncoderLayer(...)
self.transformer = nn.TransformerEncoder(...)
```

이 구조는 다음과 같은 관계를 학습하는 데 유리하다.

- 검지 끝과 손목의 관계
- 엄지와 검지의 상대적 패턴
- 손 전체 구조 내 각 관절 간 상호작용

즉 **프레임 내부의 랜드마크 관계를 보는 구조**라고 이해할 수 있다.

---

### 7-3. Pooling 방식
Transformer를 통과한 뒤 최종 feature를 만드는 방식은 다음과 같다.

```python
if self.use_cls_token:
    feat = x[:, 0]
else:
    feat = x.mean(dim=1)
```

현재 기본값은 다음과 같다.

```python
use_cls_token = False
```

즉 현재는 CLS 토큰 방식이 아니라, **랜드마크 전체 토큰의 평균(mean pooling)** 을 사용한다.

## 9. 입력 reshape 관련 사항

코드에는 다음과 같은 reshape가 있다.

```python
x = x.view(x.shape[0], self.num_landmarks, self.coord_dim)
```

이 부분은 ONNX export 시 일반적으로 `reshape` 연산으로 변환될 가능성이 높다.

즉 모델은 내부적으로 다음 두 형태를 모두 처리할 수 있는 구조일 수 있다.

다만 실제 ONNX 변환 후 입력 shape은 export 방식에 따라 달라질 수 있으므로,

## 11. 데이터 전처리 방식

### 11-1. 사용 좌표
총 63개 좌표값을 사용한다.

```python
for i in range(21):
    coord_cols.extend([f'x{i}', f'y{i}', f'z{i}'])
```

그리고 이를 다음과 같이 reshape한다.

```python
landmarks_3d = df_processed[coord_cols].to_numpy().reshape(-1, 21, 3)
```

즉 전처리 결과는 기본적으로:

- 63차원 벡터
- 또는 `(21, 3)` 형태

로 해석할 수 있다.

---

### 11-2. downsampling 여부
정리된 내용 기준으로는 **downsampling은 하지 않았고**, 다음 정규화가 적용되었다.

- position normalization
- distance normalization

```python
if origin_idx is not None:
    landmarks_3d = apply_position_normalization(landmarks_3d, origin_idx)

if scale_idx_list is not None and origin_idx is not None:
    landmarks_3d = apply_distance_normalization(landmarks_3d, origin_idx, scale_idx_list)
```

---

## 12. position normalization

position normalization은 특정 기준 랜드마크를 원점 `(0, 0, 0)` 으로 옮기는 방식이다.

```python
def apply_position_normalization(landmarks_3d: np.ndarray, origin_idx: int) -> np.ndarray:
    return landmarks_3d - landmarks_3d[:, origin_idx:origin_idx+1, :]
```

즉 기준 랜드마크 좌표를 모든 점에서 빼는 방식이다.

---

### 기준 원점은 무엇인가?
정리된 시나리오 기준으로 원점은 `wrist`이다.

```python
"origin": "wrist"
```

즉 0번 랜드마크를 원점으로 사용한다.

웹에서 동일한 전처리를 하려면 다음과 같이 하면 된다.

```text
x_i' = x_i - x_wrist
y_i' = y_i - y_wrist
z_i' = z_i - z_wrist
```

---

## 13. distance normalization (scale normalization)

distance normalization은 기준 두 랜드마크 사이 거리를 1이 되도록 전체 좌표를 나누는 방식이다.

```python
def apply_distance_normalization(landmarks_3d: np.ndarray, origin_idx: int, scale_idx_list: list) -> np.ndarray:
    if not scale_idx_list:
        return landmarks_3d

    scale_idx = scale_idx_list[0] if len(scale_idx_list) == 1 else scale_idx_list[1]
    dist = np.linalg.norm(landmarks_3d[:, scale_idx, :] - landmarks_3d[:, origin_idx, :], axis=1)
    safe_dist = np.maximum(dist, 1e-6)
    return landmarks_3d / safe_dist[:, np.newaxis, np.newaxis]
```

---

### scale 기준은 무엇인가?
정리된 시나리오 기준으로 scale은 다음과 같다.

```python
"scale": ["wrist", "middle_mcp"]
```

즉 `wrist(0)` 와 `middle_mcp(9)` 사이의 거리를 기준으로 정규화한다.

실제 의미는 다음과 같다.

1. 먼저 wrist 기준으로 위치 정렬
2. `wrist ↔ middle_mcp` 거리 계산
3. 모든 좌표를 그 거리로 나눔

예시는 다음과 같다.

```python
scale = distance(landmark[0], landmark[9])
normalized = normalized / scale
```

## 14. pos_scale 시나리오의 정확한 의미

정리된 설정은 다음과 같다.

```python
"pos_scale": {
    "downsample_ratio": "origin",
    "origin": "wrist",
    "scale": ["wrist", "middle_mcp"],
    "augment": False
}
```

이 설정이 의미하는 바는 다음과 같다.

- downsampling 없음으로 이해됨
- 위치 정규화 기준: `wrist`
- 스케일 정규화 기준: `wrist ↔ middle_mcp`
- augmentation 없음

즉 웹에서 같은 입력을 만들려면 **반드시 동일한 정규화 절차를 따라야 한다.**

## 16. JOINT_COLS와 RAW_JOINT_COLS

### 16-1. JOINT_COLS의 정확한 순서
정리된 내용 기준으로 `JOINT_COLS`는 다음과 같다.

```python
JOINT_COLS = [f"n{axis}{i}" for i in range(21) for axis in ("x", "y", "z")]
```

즉 순서는 다음과 같이 확정된다.

- `nx0, ny0, nz0`
- `nx1, ny1, nz1`
- ...
- `nx20, ny20, nz20`

즉 **정규화된 좌표 컬럼**을 사용한다는 뜻이다.

---

### 16-2. RAW_JOINT_COLS의 순서
원본 좌표 컬럼은 다음과 같다.

```python
RAW_JOINT_COLS = [f"{axis}{i}" for i in range(21) for axis in ("x", "y", "z")]
```

즉 순서는 다음과 같다.

- `x0, y0, z0`
- `x1, y1, z1`
- ...
- `x20, y20, z20`

---

### 16-3. 중요한 결론
이 모델이 사용하는 것은 raw 좌표가 아니라 **normalized joint 컬럼**이다.

즉:

- 학습 입력: `JOINT_COLS` 기반
- raw 컬럼: `RAW_JOINT_COLS`
- 실제 모델 입력용 feature는 정규화된 좌표

따라서 웹에서도 raw 좌표를 그대로 넣는 것이 아니라,  
**학습 때와 동일한 정규화 결과 좌표를 넣어야 한다.**

## 17. 공통 프레임 데이터 변환 방식

정리된 코드 기준으로 공통 변환 함수는 다음과 같다.

```python
def frame_arrays(df, feature_cols):
    X = df[feature_cols].to_numpy(dtype=np.float32)
    y = df["gesture"].to_numpy(dtype=np.int64)
    meta = [sample_metadata(df, i) for i in range(len(df))]
    return X, y, meta
```

여기서 알 수 있는 점은 다음과 같다.

- `X`: feature 배열
- `y`: gesture 라벨
- `meta`: 샘플 메타데이터

---

### FrameDataset의 기본 반환 형태
정리된 내용에 따르면 공통 `FrameDataset`은 기본적으로 `(D,)` 벡터를 반환한다.

즉 프레임 하나를 **1차원 feature 벡터**로 다루는 공통 구조가 있으며,  
모델 내부 또는 별도 단계에서 `(21, 3)` 형태로 reshape하는 구조로 해석할 수 있다.

## 18. 입력 좌표의 원천: MediaPipe

정리된 내용에 따르면 원본 좌표는 **MediaPipe Hand Landmarker**에서 추출된다.

즉 전체 파이프라인의 출발점은 다음과 같다.

1. 비디오 프레임
2. MediaPipe Hand Landmarker
3. 21개 손 랜드마크 추출
4. 각 랜드마크의 `x, y, z` 사용

---

## 19. 사용하는 좌표계: world 좌표가 아니라 normalized landmarks

정리된 내용에서 매우 중요한 부분은 다음이다.

- `result.hand_landmarks` 사용
- `result.hand_world_landmarks`는 사용하지 않음

즉 학습 데이터의 `x0, y0, z0 ... x20, y20, z20`는  
**MediaPipe의 normalized landmark 좌표**를 기반으로 한다는 뜻이다.

이 말은 곧 웹에서도 다음이 중요하다는 뜻이다.

- `worldLandmarks`를 쓰면 안 됨
- 학습 때와 같은 종류의 좌표계를 써야 함
- 즉 **normalized hand landmarks**를 사용해야 함

## 21. CSV 한 행의 구조와 메타 정보

정리된 내용 기준으로 CSV 한 행은 대략 다음과 같은 구조로 만들어진다.

```python
row = [frame_idx, format_timestamp(frame_idx, actual_fps), None] + landmark_...
```

즉 메타 정보로 다음이 함께 관리된다.

- `frame_idx`
- `timestamp`

이 정보는 웹 실시간 추론 자체에 직접 필수는 아닐 수 있지만,  
다음 용도로 유용하다.

- 디버깅
- 영상과 결과 싱크 확인
- 특정 프레임 위치 추적

## 22. 체크포인트 복원 관련 정보

정리된 내용에 따르면 체크포인트에서 다음 정보는 일부 추론 가능하다.

- `num_classes`
- `use_cls_token` 여부
- `d_model`
- classifier weight shape 등

또한 `Landmark_Spatial_Transformer`는 입력 feature를 사실상 63차원으로 보고 복원하는 흐름으로 이해할 수 있다.

다만 체크포인트 복원 로직은 실제 구현 코드에 따라 달라질 수 있으므로,  
이 부분은 **실제 로딩 코드 기준으로 최종 확인**이 필요하다.

## 23. 웹 배포 관점에서 최종 핵심 정리

웹에서 이 모델을 정확히 재현하려면 아래 조건을 맞춰야 한다.

### 필수 조건
1. MediaPipe의 **normalized hand landmarks** 사용
2. 랜드마크 21개 사용
3. 각 랜드마크의 `(x, y, z)` 사용
4. 랜드마크 순서를 학습 때와 동일하게 유지
5. `wrist` 기준 position normalization 수행
6. `wrist ↔ middle_mcp` 거리 기준 scale normalization 수행
7. 최종적으로 정규화된 63개 값을 모델 입력으로 사용
8. 입력 dtype은 `float32`
9. 출력은 확률이 아니라 `(1, 7)` 형태의 logits

## 25. 주의할 점

다음 항목은 특히 실수하기 쉽다.

### 25-1. shape만 맞추면 되는 것이 아님
`(63,)` 또는 `(21, 3)` shape만 맞춘다고 끝이 아니다.  
**랜드마크 순서와 정규화 방식까지 동일해야 한다.**

### 25-2. raw 좌표와 normalized 좌표를 혼동하면 안 됨
이 모델은 raw 좌표가 아니라 **정규화된 좌표 기반 입력**을 사용한다.  
웹에서도 같은 전처리를 거친 값을 넣어야 한다.

### 25-3. world landmarks를 사용하면 안 됨
학습 데이터 기준과 다를 수 있으므로,  
동일 재현을 위해서는 **MediaPipe normalized landmarks**를 사용해야 한다.

### 25-4. 출력은 확률이 아니다
출력은 softmax 이전의 logits이다.  
웹에서 확률처럼 바로 쓰면 안 된다.

이 모델은 **MediaPipe normalized hand landmarks 21개를 wrist 기준 위치 정규화와 wrist-middle_mcp 기준 스케일 정규화 후 입력받아, 프레임 단위로 7개 클래스의 logits를 출력하는 단일 프레임 Transformer 기반 손 제스처 분류 모델**이다.
