# JamJamBeat 모델 검증 파이프라인 해설

본 문서는 잼잼비트 제스처 분류기의 검증 파이프라인이 어떤 철학과 흐름으로 구성되어 있는지 설명하는 오버뷰(Overview) 문서입니다.

현재 실제 구현은 아래 경로를 중심으로 동작합니다.

- 학습/평가 실행: `model/model_pipelines/run_pipeline.py`
- 평가 산출물 생성: `model/model_evaluation/모델검증관련파일/evaluation_runtime.py`
- 평가 결과 사용 가이드: `model/model_evaluation/모델검증관련파일/evaluation_guide.md`
- 추론/뷰어 계열 확인 도구: `model/model_evaluation/모델별영상체크/`

즉 이 문서는 설계 철학과 전체 흐름을 설명하고, 실제 결과 파일을 읽는 법은 `evaluation_guide.md`가 맡는 구조로 보면 됩니다.

---

## 🚀 전체 검증 파이프라인 5단계 흐름

### 1단계: 데이터 입력 및 추론 로깅 (Predict Wrapper 구축)
* **참고 위치**: `classifier_output_and_visualization.md`의 **2. 추론 래퍼 코드 (`predict_with_log`)**
* **핵심 역할**: 파이프라인의 시작점입니다. 랜드마크 데이터를 단순히 분류기에 넣고 결과만 빼는 것이 아니라, 시스템 성능의 해부학적 단서들을 모두 기록합니다.
* **로깅 내용**:
  * **각 클래스별 세부 확률** ($p_0 \sim p_6$)
  * **단계별 지연 시간** (MediaPipe 추출, Feature 정규화, MLP 추론, Postprocess 후처리 시간 각각 분리 측정)
* **산출물**: 이 모든 정보가 담긴 풍부한 `preds.csv` 파일. (이후 모든 평가의 기준 데이터가 됩니다.)

### 2단계: 기본 분류 성능 평가 (프레임 단위)
* **참고 위치**: `evaluation_guide.md`의 **1. 기본 평가 — Confusion Matrix + F1-score**
* **핵심 역할**: 모델의 원초적인 분류 능력을 파악합니다.
* **분석 내용**: 만들어진 `preds.csv`를 정답 라벨(GT)과 비교하여, 모델이 어떤 손 모양을 어떤 모양으로 헷갈려 하는지 2D 히트맵(Confusion Matrix)으로 시각화하고, 각 클래스별 정밀도(Precision)와 재현율(Recall)을 확인합니다.

### 3단계: 실사용 체감 품질 방어 (오탐 제어 및 FP/min 측정)
* **참고 위치**: `evaluation_guide.md`의 **2. 응용 평가 ① — FP/min (오탐 트리거 빈도)**
* **핵심 역할**: 이 프로젝트의 최우선 과제인 **"잘못된 소리(오탐)가 나서 UX를 망치는 일"**을 막기 위한 실무적 평가입니다.
* **분석 내용**: 모델의 날것(Raw) 확률 예측 결과에 실제 앱과 동일한 필터 세트(**Threshold 임계값 $\rightarrow$ Voting 다수결 $\rightarrow$ Debounce 히스테리시스**)를 걸어 이벤트를 생성해 봅니다. 이후 아무 제스처도 아닌 평상시(Neutral) 동작에서 1분에 오류가 몇 번이나 나는지(`FP/min`)를 엄격하게 계산합니다.

### 4단계: 실시간성 및 병목 구간 분석 (지연/지터 평가)
* **참고 위치**: `evaluation_guide.md`의 **3. 응용 평가 ② — 트리거 지연 + Jitter CDF**
* **핵심 역할**: 시스템이 사용자 움직임에 얼마나 즉각적이고 안정적으로 반응하는지(리듬감)를 평가합니다.
* **분석 내용**: 
  * 1단계에서 측정한 단계별 지연 시간을 **Box Plot**으로 그려 어느 모듈이 느린지(병목) 찾습니다.
  * 전체 지연 시간을 **Latency CDF 플롯**으로 그려 시스템이 간헐적으로 멈추는지(Spike) 확인합니다.
  * 이벤트 간의 변동성(**Jitter**)을 측정하여 실제 사운드 출력의 리듬감이 흔들리지 않는지 점검합니다.

### 5단계: 과적합 및 신뢰도 심화 검증 (고급 시각화)
* **참고 위치**: `classifier_output_and_visualization.md`의 **5.5 ~ 5.7 섹션**
* **핵심 역할**: 분류기 모델 내부의 건전성을 심층 진단합니다.
* **분석 내용**:
  * **Learning Curve (학습 곡선)**: Train/Val Loss를 비교해 과적합(Overfitting) 여부를 진단.
  * **Temporal Jitter (시간축 확률 변동)**: 동작이 전환되는 시점에서 모델의 확률이 얼마나 요동치는지(Flickering) 시각화하여 디바운싱(Debounce) K값을 설정할 단서 제공.
  * **Reliability Diagram (신뢰도 교정)**: 모델이 90% 확신한다고 뱉은 결과가 실제로도 90% 맞는지를 검증하여 모델의 '과잉 확신' 상태를 점검.

## 🧭 현재 구현 기준 구조 요약

위 5단계 철학은 그대로 유지되지만, 현재 프로젝트의 실제 동작 방식은 아래처럼 정리되어 있습니다.

### 1. 입력 데이터 역할 분리

현재 `run_pipeline.py`는 역할이 나뉜 CSV를 직접 받습니다.

- `--train-csv`
- `--val-csv`
- `--test-csv`
- `--inference-csv`

이 중 공식 비교와 랭킹은 `*_test.csv`를 기준으로 하고, `*_inference.csv`는 별도 hold-out 확인용으로 사용합니다.

### 2. 공식 test와 inference의 해석 차이

현재 정책은 아래로 고정되어 있습니다.

- 공식 `test`:
  - `static_images_63d`
  - sequence 모델은 `independent_repeat`
- `inference`:
  - 기존 sliding 해석 유지

즉 sequence 모델도 공식 랭킹에서는 정지사진 기반 `*_test.csv`를 기준으로 비교하고, 시간축 동작 확인은 `preds_inference.csv`나 뷰어에서 따로 봅니다.

### 3. 실제 산출물 위치

run 결과는 보통 아래 경로에 저장됩니다.

```text
model/model_evaluation/pipelines/{suite_name}/{model_id}/{run_id}/
```

핵심 파일:

- `preds_test.csv`
- `preds_inference.csv`
- `run_summary.json`
- `evaluation/`

`evaluation/` 아래에는 현재 아래 결과가 생성됩니다.

- 클래스 기준:
  - `confusion_matrix.csv`
  - `confusion_matrix.png`
  - `per_class_report.csv`
  - `per_class_summary.csv`
  - `per_class_misclassifications.csv`
- 테스트케이스(`source_file`) 기준:
  - `per_test_case_report.csv`
  - `per_test_case_misclassifications.csv`
  - `test_case_accuracy.png`
  - `test_case_confusion.png`
- 공통 메타:
  - `metrics_summary.json`
  - `dataset_info.json`
  - `latency_cdf.png`

이미지 평가 경로에서는 여기에 더해 파일명 기반 포즈케이스(`BaseP`, `RollP`, `PitchP`, `YawP`, `NoneNetural`, `NoneOther`) 집계도 추가될 수 있습니다.

## 💡 요약: 개발자 적용 가이드

이 파이프라인의 설계 사상은 **"단순 정확도(Accuracy)를 넘어서, 실제 아동 사용자가 느끼는 오류(오탐)와 버벅임(지연)을 철저히 잡아내는 것"**입니다. 

현재 구현 기준으로는 별도 `src/training/*`를 새로 만드는 방식보다, 아래 흐름으로 이해하는 것이 정확합니다.

- `run_pipeline.py`가 학습과 예측 CSV 생성을 담당
- `evaluation_runtime.py`가 평가 결과 CSV/PNG/JSON 생성을 담당
- `evaluation_guide.md`가 실제 산출물 읽는 법을 담당

즉 이 문서는 **설계 철학과 단계별 의미를 설명하는 오버뷰**로 읽으면 됩니다.
