# 재현성·인과해석 리서치

이 문서는 JamJamBeat 현재 파이프라인에서 이미 남고 있는 로그를 기준으로, 연구 재현성과 인과적 해석에 실제로 도움이 되는 항목과 아직 부족한 항목을 정리한다.

이번 정리는 실행 환경 정보나 원시 예측 CSV 경로보다, 데이터셋 규모·전처리·학습 보정 요소·평가 후처리처럼 결과 해석에 직접 영향을 주는 로그에 집중한다.

## 1. 현재 로그 체계 개요

프로젝트는 크게 네 단계에서 메타데이터를 남긴다.

| 단계 | 주요 산출물 | 현재 해석 가능한 것 | 한계 |
|---|---|---|---|
| dataset build | `dataset_manifest.csv`, `source_split_manifest.json` | 데이터셋별 row 수, split별 source 수, normalization family, train/test 원본 CSV 연결 | 라벨 분포와 전처리 수식은 없음 |
| suite 실행 | `comparison_suite.json`, `comparison_results.csv` | 어떤 dataset key로 어떤 모델 조합을 돌렸는지, suite 단위 training recipe | 모델별 상세 학습 로그는 없음 |
| 개별 run | `run_summary.json`, `train_history.csv`, `model.pt` | 하이퍼파라미터, split 규모, dataset info, checkpoint 검증, epoch별 LR 변화 | scheduler 이름/정책이 명시적이지 않음 |
| evaluation | `evaluation/` 하위 CSV/PNG/JSON | confusion, per-class report, latency, class0 지표 | train/val 분포나 전처리 의도 자체는 설명하지 않음 |

핵심은 새 파이프라인이 `dataset build -> suite -> run -> evaluation` 순으로 메타를 나누어 남긴다는 점이다. 연구 관점에서는 이 체계를 따라 로그를 읽어야 원인을 단계별로 분리할 수 있다.

## 2. 현재 이미 기록되는 정보

### 2.1 데이터셋 규모와 split 규모

현재 파이프라인은 데이터셋 규모를 비교적 잘 남긴다.

- `dataset_manifest.csv`
  - `dataset_key`
  - `normalization_family`
  - `split`
  - `source_count`
  - `row_count`
- `source_split_manifest.json`
  - train / val / inference source 목록
  - inference source map
- `run_summary.json`
  - `split_sizes`
  - `dataset_sizes`
  - `dataset_info.source_counts`
  - `dataset_info.rows_by_source_group`

즉 현재는 “어떤 시나리오가 더 큰 데이터셋이었는가”, “어느 split이 몇 개 source와 몇 개 row로 구성됐는가”는 재구성 가능하다.

### 2.2 normalization family와 dataset key

현재 런에서는 아래 두 항목이 suite/run 모두에 기록된다.

- `dataset_key`
- `normalization_family`

그래서 `baseline`, `pos_only`, `scale_only`, `pos_scale`의 차이를 결과와 연결하는 것은 가능하다. 다만 “정규화가 수학적으로 어떻게 적용됐는가”는 별도 문서나 코드 독해가 필요하다.

### 2.3 학습 하이퍼파라미터

`run_summary.json.hyperparameters`에는 아래가 기록된다.

- `epochs`
- `batch_size`
- `lr`
- `optimizer`
- `weight_decay`
- `patience`
- `loss_type`
- `use_weighted_sampler`
- `use_alpha`
- `use_label_smoothing`
- `focal_gamma`
- `label_smoothing`
- `sampler_policy`
- `alpha_policy`
- `tau`
- `vote_n`
- `debounce_k`
- `fallback_fps`

이 수준이면 현재 실험에서 **어떤 손실함수 계열과 어떤 보정 요소가 켜져 있었는지**는 꽤 명확하게 재구성 가능하다.

### 2.4 optimizer / learning rate / scheduler

현재 코드에서 optimizer는 `AdamW`로 고정이다. 이 정보는 `run_summary.json.hyperparameters.optimizer`에 기록된다.

learning rate는 두 방식으로 남는다.

- 초기값: `run_summary.json.hyperparameters.lr`
- epoch별 변화: `train_history.csv`의 `lr` 컬럼

scheduler는 코드상 `CosineAnnealingLR`가 고정이지만, 현재 메타에 scheduler 이름이나 파라미터가 직접 기록되지는 않는다. 대신 `train_history.csv`의 `lr` 변화 곡선을 통해 간접적으로만 확인된다.

### 2.5 checkpoint verification

새 run에서는 `run_summary.json.checkpoint_verification`에 아래가 포함된다.

- checkpoint 경로
- fingerprint
- strict reload 결과
- saved/stored state 일치 여부
- missing/unexpected key 여부

이 덕분에 “성능 차이가 checkpoint 로딩 문제 때문인지”를 의심할 때 최소한의 검증 근거는 이미 남는다.

## 3. 현재 빠져 있거나 약한 정보

### 3.1 전체 / role별 라벨 분포

지금 가장 크게 비는 항목이다.

현재는 row 수와 source 수는 잘 남지만,

- 전체 클래스 분포
- train / val / test / inference split별 클래스 분포
- source별 클래스 분포 요약

은 정식 메타 파일에 없다.

이 정보가 없으면 `macro_f1` 변화가 실제로 class imbalance 완화 때문인지, 단순히 split 라벨 조성이 바뀐 것인지 해석하기 어렵다.

### 3.2 전처리 세부 변환 설명

현재 `normalization_family`와 `dataset_key`는 남지만,

- pos_only가 어떤 좌표 이동을 의미하는지
- scale_only가 어떤 스케일 보정을 의미하는지
- pos_scale이 두 보정을 어떤 순서로 적용하는지
- raw joint / normalized joint / bone / angle feature가 어떤 조합인지

는 로그에서 직접 읽을 수 없다.

즉 “dataset variant가 달랐다”는 사실은 알 수 있지만, “무엇이 실제로 달랐는가”는 로그만으로는 충분히 설명되지 않는다.

### 3.3 손실함수 보정 요소의 해석용 요약

현재 메타는 `use_alpha`, `use_weighted_sampler`, `use_label_smoothing`, `loss_type`을 기록한다.  
하지만 연구 해석용으로는 아래가 별도 설명 없이 바로 읽히지 않는다.

- alpha가 train label 분포로부터 inverse-frequency 방식으로 계산된다는 점
- sampler도 동일하게 inverse-frequency 가중을 쓴다는 점
- CE와 Focal 모두 alpha를 받을 수 있다는 점
- smoothing 값이 `_shared.py` 상수에서 온다는 점

즉 “스위치 상태”는 기록되지만, “그 스위치가 실제 수학적으로 무엇을 의미하는가”는 로그 바깥 설명이 필요하다.

### 3.4 과거 run과의 메타 불연속

`20260313` 계열 과거 run은 현재 run에 비해 메타가 단순하다.

대표적으로 과거 run은 아래가 빠지거나 약하다.

- `dataset_key`
- `normalization_family`
- suite-level `training_recipe`
- `loss_type / use_alpha / use_weighted_sampler / use_label_smoothing`
- `checkpoint_verification`
- `tau / vote_n / debounce_k / fallback_fps`

그래서 과거 결과와 현재 결과를 한 표에서 직접 비교할 때는, 성능 차이보다 먼저 **메타 해상도 차이**를 고려해야 한다.

## 4. 학습 보정 요소 분석

### 4.1 현재 기본값

`_shared.py`의 기본 상수는 다음과 같다.

| 항목 | 현재 기본값 |
|---|---|
| `DEFAULT_LOSS_TYPE` | `focal` |
| `DEFAULT_USE_WEIGHTED_SAMPLER` | `True` |
| `DEFAULT_USE_ALPHA` | `True` |
| `DEFAULT_USE_LABEL_SMOOTHING` | `False` |
| `DEFAULT_FOCAL_GAMMA` | `2.0` |
| `DEFAULT_LABEL_SMOOTHING` | `0.1` |

즉 코드 기본값만 보면 현재 기본 레시피는 `Focal + WeightedSampler + alpha + no smoothing`이다.

### 4.2 실제 학습 조립 규칙

`run_pipeline.py`의 `build_training_recipe()`는 아래 순서로 학습 보정 요소를 조립한다.

1. `use_weighted_sampler=True`
   - `create_weighted_sampler(labels)` 호출
   - class inverse-frequency 기반 `WeightedRandomSampler` 생성
   - 이 경우 train loader는 `shuffle=False`
2. `use_alpha=True`
   - `compute_alpha(labels, num_classes, device)` 호출
   - train label 분포 기반 inverse-frequency 정규화 alpha 생성
3. `use_label_smoothing=True`
   - `DEFAULT_LABEL_SMOOTHING` 값 적용
4. `loss_type`
   - `cross_entropy`면 `CrossEntropyLoss(weight=alpha, label_smoothing=...)`
   - `focal`이면 `FocalLoss(alpha=alpha, gamma=focal_gamma, label_smoothing=...)`

즉 현재 구조에서 `sampler`와 `alpha`는 서로 독립 스위치다. 둘을 동시에 켤 수도 있고, 하나만 켤 수도 있다.

### 4.3 optimizer / scheduler

현재 학습 코드는 아래로 고정되어 있다.

- optimizer: `AdamW`
- scheduler: `CosineAnnealingLR`

이 구조는 현재 run마다 바뀌지 않기 때문에, 결과 해석 시 “optimizer/scheduler를 바꾼 실험”은 아직 아니다.  
다만 `train_history.csv`의 `lr` 컬럼 덕분에 scheduler가 실제로 epoch마다 learning rate를 바꿨다는 사실은 확인 가능하다.

### 4.4 인과 해석상 주의점

학습 보정 요소는 데이터셋 효과와 강하게 얽힌다.

특히 아래 조합은 해석을 어렵게 만든다.

- normalization family 변경
- loss_type 변경
- weighted sampler on/off
- alpha on/off
- label smoothing on/off
- tau 같은 평가 후처리 변경

이 항목들이 한 번에 바뀌면, 성능 변화가

- 데이터 전처리 때문인지
- class imbalance 완화 때문인지
- overconfidence 완화 때문인지
- 후처리 threshold 때문인지

분리해 말하기 어렵다.

그래서 연구 해석에서는 최소한

- dataset variant
- loss recipe
- post-processing

세 축을 분리해 기록하고 비교해야 한다.

## 5. 과거 런과 현재 런 차이

### 5.1 과거 런 (`20260313` 계열)

과거 run은 다음 특징이 있다.

- 입력 CSV 경로는 남지만 dataset key 수준의 의미 요약이 약함
- loss recipe가 명시적으로 남지 않음
- checkpoint verification이 없음
- suite 메타가 dataset tag와 모델 목록 위주임

즉 과거 run은 “무엇을 돌렸는가”는 대략 알 수 있지만, “어떤 학습 보정이 들어갔는가”까지는 부족하다.

### 5.2 현재 런 (`20260319` 계열)

현재 run은 아래가 보강되었다.

- `dataset_key`
- `normalization_family`
- `training_recipe`
- `loss_type / use_weighted_sampler / use_alpha / use_label_smoothing`
- `focal_gamma / label_smoothing`
- `tau / vote_n / debounce_k / fallback_fps`
- `checkpoint_verification`

즉 현재 런은 과거보다 재현성과 해석 가능성이 훨씬 높아졌다.  
다만 라벨 분포와 전처리 세부 의미는 여전히 메타 바깥 설명이 필요하다.

## 6. 연구용 권고안

### 6.1 지금 문서만으로 해석 가능한 것

- 어떤 dataset variant를 사용했는가
- 어느 정도 규모의 split이었는가
- 어떤 loss recipe를 사용했는가
- sampler / alpha / smoothing이 켜졌는가
- optimizer와 learning rate 흐름이 어땠는가
- checkpoint가 일관되게 저장/재로딩됐는가

### 6.2 지금 문서만으로는 부족한 것

- class imbalance가 실제로 split마다 어떻게 달랐는가
- 전처리 variant의 수학적 차이가 무엇인가
- 특정 성능 향상이 dataset 효과인지 loss 효과인지 명확히 분리되는가

### 6.3 다음 로그 보강 우선순위

우선순위는 아래가 적절하다.

1. split별 라벨 분포
2. normalization/feature schema 설명 메타
3. scheduler 이름/설정 명시
4. loss recipe의 해석용 요약 필드

이 네 항목이 보강되면, 현재 프로젝트는 “성능이 좋아졌다” 수준을 넘어 “왜 좋아졌는가”를 훨씬 안정적으로 말할 수 있게 된다.
