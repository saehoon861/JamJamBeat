# 학습 보정 요소 정리

이 문서는 설정 사용법이 아니라, **현재 코드가 실제로 loss / alpha / sampler / smoothing / optimizer / scheduler를 어떻게 조립하는지**를 연구자 관점에서 정리한 메모다.

## 1. 기본값 요약

현재 `_shared.py`의 상수 기준 기본값은 아래와 같다.

| 항목 | 기본값 |
|---|---|
| `DEFAULT_LOSS_TYPE` | `focal` |
| `DEFAULT_USE_WEIGHTED_SAMPLER` | `True` |
| `DEFAULT_USE_ALPHA` | `True` |
| `DEFAULT_USE_LABEL_SMOOTHING` | `False` |
| `DEFAULT_FOCAL_GAMMA` | `2.0` |
| `DEFAULT_LABEL_SMOOTHING` | `0.1` |

즉 “기본 코드 상태”만 보면 `Focal + WeightedSampler + alpha + no smoothing`이 기본 레시피다.

중요한 점은, 실제 실험은 이 기본값과 다를 수 있다는 점이다.  
예를 들어 `20260319` 계열 일부 suite는 `CrossEntropy + WeightedSampler + no alpha + label_smoothing=0.1`로 실행되었고, 이 값은 `comparison_suite.json.training_recipe`와 `run_summary.json.hyperparameters`에 남는다.

## 2. 실제 학습 조립 규칙

현재 `run_pipeline.py`의 `build_training_recipe()`는 아래 규칙으로 작동한다.

### 2.1 weighted sampler

- `use_weighted_sampler=True`
  - `create_weighted_sampler(labels)` 사용
  - class inverse-frequency 기반 `WeightedRandomSampler` 생성
  - train loader는 sampler를 쓰고 `shuffle=False`
- `use_weighted_sampler=False`
  - sampler 없이 `shuffle=True`

즉 sampler는 class imbalance를 minibatch sampling 단계에서 보정하는 역할이다.

### 2.2 alpha

- `use_alpha=True`
  - `compute_alpha(labels, num_classes, device)` 사용
  - train label의 inverse-frequency를 정규화한 alpha 계산
- `use_alpha=False`
  - loss weight 없음

alpha는 loss 내부에서 클래스별 가중치 역할을 한다.

### 2.3 loss type

- `loss_type="cross_entropy"`
  - `CrossEntropyLoss(weight=alpha, label_smoothing=...)`
- `loss_type="focal"`
  - `FocalLoss(alpha=alpha, gamma=focal_gamma, label_smoothing=...)`

즉 alpha는 CE와 Focal 양쪽에 모두 들어갈 수 있다.

### 2.4 label smoothing

- `use_label_smoothing=True`
  - `DEFAULT_LABEL_SMOOTHING` 적용
- `use_label_smoothing=False`
  - `0.0`

Focal에서도 smoothing은 CE 항 계산에 같이 들어간다.

## 3. optimizer / scheduler

현재 학습 루프는 아래로 고정이다.

- optimizer: `AdamW`
- scheduler: `CosineAnnealingLR`

이 점은 해석상 중요하다. 현재 실험 결과 차이는 optimizer 탐색 때문이 아니라,

- dataset variant
- architecture
- loss recipe
- 평가 후처리

의 조합 차이에서 오는 경우가 대부분이다.

또한 scheduler는 메타에 직접 이름이 기록되지는 않지만, `train_history.csv`의 `lr` 컬럼을 보면 cosine decay가 실제로 적용된 흔적을 볼 수 있다.

## 4. 현재 메타 반영 위치

### 4.1 suite-level

`comparison_suite.json.training_recipe`에는 다음이 남는다.

- `loss_type`
- `use_weighted_sampler`
- `use_alpha`
- `use_label_smoothing`
- `focal_gamma`
- `label_smoothing_default`

즉 suite 단위에서는 “이번 배치가 어떤 보정 정책으로 돌았는가”를 알 수 있다.

### 4.2 run-level

`run_summary.json.hyperparameters`에는 다음이 남는다.

- `loss_type`
- `use_weighted_sampler`
- `use_alpha`
- `use_label_smoothing`
- `focal_gamma`
- `label_smoothing`
- `sampler_policy`
- `alpha_policy`
- `optimizer`
- `lr`
- `weight_decay`
- `patience`

즉 개별 모델 run 단위에서는 suite보다 조금 더 자세하게 “실효 설정”을 읽을 수 있다.

## 5. 해석상 주의점

### 5.1 sampler + alpha 동시 사용

현재 구조는 sampler와 alpha를 동시에 켤 수 있다.  
이 경우 class imbalance를

- sampling 단계
- loss 단계

두 군데에서 동시에 보정한다.

연구 해석에서는 이 조합을 “이중 보정”으로 볼 수 있다. 따라서 성능 변화가 좋아졌거나 나빠졌을 때, 그 원인이 sampler인지 alpha인지 분리하기 어려워진다.

### 5.2 focal과 smoothing의 조합

Focal은 쉬운 샘플의 loss 기여를 줄이고, smoothing은 예측 확신을 낮추는 방향으로 작동한다.  
둘은 동시에 쓸 수 있지만, 해석 관점에서는 다음을 같이 봐야 한다.

- class0_fnr 감소가 smoothing 때문인지
- 희소 클래스 recall 변화가 focal 때문인지
- 둘의 조합이 neutral decision boundary를 어떻게 바꿨는지

즉 `focal + smoothing`은 가능하지만, 단독 효과 분리는 더 어렵다.

### 5.3 dataset 효과와 loss 효과 분리 문제

현재 결과표를 비교할 때 아래가 동시에 바뀌면 인과 해석이 급격히 어려워진다.

- normalization family
- loss_type
- use_alpha
- use_weighted_sampler
- use_label_smoothing
- tau

예를 들어 `macro_f1`이 올랐더라도 그 이유가

- pos_scale 전처리 때문인지
- CE로 바꿔서 그런지
- sampler만 유지하고 alpha를 꺼서 그런지
- tau 조정 때문인지

분리해서 말하기 어렵다.

따라서 연구 관점에서는 최소한 아래 세 축은 분리 비교가 필요하다.

1. dataset variant
2. training recipe
3. evaluation 후처리

## 6. 현재 코드 기준 해석 포인트

현재 프로젝트에서 학습 보정 요소를 볼 때 가장 중요한 해석 포인트는 아래다.

- 기본값과 실제 실행 recipe는 다를 수 있다.
- `comparison_suite.json`은 suite 단위 recipe 요약을 준다.
- `run_summary.json`은 개별 run의 실효 하이퍼파라미터를 준다.
- `train_history.csv`는 scheduler 효과를 간접적으로 보여준다.

즉 현재는 “보정 요소가 켜졌는지”는 충분히 남지만, “그 보정이 실제 분포에 어떤 영향을 주었는지”를 보려면 여전히 라벨 분포 로그가 추가로 필요하다.
