# 현재 로그 커버리지 체크리스트

이 표는 현재 프로젝트가 연구 해석에 필요한 항목을 어느 정도까지 기록하고 있는지 빠르게 점검하기 위한 것이다.

상태 기준:

- `충분`: 현재 메타만으로 해석 가능
- `부분`: 일부는 알 수 있으나 추가 설명이나 코드 독해가 필요
- `없음`: 현재 공식 메타에서 직접 확인하기 어려움

| 항목 | 현재 기록 위치 | 현재 상태 | 연구 해석 관점 메모 |
|---|---|---|---|
| 데이터셋 규모 | `dataset_manifest.csv`, `run_summary.json.dataset_info.total_rows` | 충분 | dataset별 전체 규모와 run 단위 총 row 수를 확인할 수 있다. |
| split별 source count | `dataset_manifest.csv`, `source_split_manifest.json`, `comparison_suite.json`, `run_summary.json.dataset_info.source_counts` | 충분 | train/val/inference/test의 source 수 비교가 가능하다. |
| split별 row count | `dataset_manifest.csv`, `run_summary.json.split_sizes`, `run_summary.json.dataset_info.split.*.rows` | 충분 | split 규모 차이에 따른 성능 해석이 가능하다. |
| normalization family | `comparison_suite.json`, `run_summary.json` | 충분 | `baseline / pos_only / scale_only / pos_scale` 수준 비교는 가능하다. |
| dataset key | `comparison_suite.json`, `run_summary.json`, `dataset_manifest.csv` | 충분 | scenario 단위 식별은 안정적이다. |
| 전처리 유형 | `dataset_key`, `normalization_family` | 부분 | 어떤 family인지는 알 수 있지만 수학적 변환 규칙은 별도 문서가 필요하다. |
| 전체 라벨 분포 | 공식 메타 없음 | 없음 | class imbalance를 직접 읽을 수 없어 loss 효과 해석이 제한된다. |
| split별 라벨 분포 | 공식 메타 없음 | 없음 | train/val/test/inference 조성 차이를 결과와 바로 연결하기 어렵다. |
| optimizer | `run_summary.json.hyperparameters.optimizer` | 충분 | 현재는 `AdamW`가 명시적으로 남는다. |
| learning rate | `run_summary.json.hyperparameters.lr`, `train_history.csv.lr` | 충분 | 초기값과 epoch별 변화 둘 다 확인 가능하다. |
| scheduler | 코드, `train_history.csv.lr` | 부분 | 현재는 lr 궤적은 남지만 scheduler 이름/설정은 메타에 직접 없다. |
| loss_type | `comparison_suite.json.training_recipe`, `run_summary.json.hyperparameters.loss_type` | 충분 | CE/Focal 구분은 안정적으로 남는다. |
| alpha 사용 여부 | `comparison_suite.json.training_recipe`, `run_summary.json.hyperparameters.use_alpha`, `alpha_policy` | 충분 | on/off는 명확하다. 계산 방식은 코드 설명이 필요하다. |
| weighted sampler 사용 여부 | `comparison_suite.json.training_recipe`, `run_summary.json.hyperparameters.use_weighted_sampler`, `sampler_policy` | 충분 | on/off는 명확하다. sampling weight 정의는 코드 설명이 필요하다. |
| label smoothing 사용 여부 | `comparison_suite.json.training_recipe`, `run_summary.json.hyperparameters.use_label_smoothing`, `label_smoothing` | 충분 | 활성화 여부와 실효 값이 남는다. |
| focal gamma | `comparison_suite.json.training_recipe.focal_gamma`, `run_summary.json.hyperparameters.focal_gamma` | 충분 | gamma 값 자체는 남는다. |
| checkpoint verification | `run_summary.json.checkpoint_verification` | 충분 | 새 run은 fingerprint와 strict reload 결과를 비교 가능하다. 과거 run은 부족하다. |
| evaluation 후처리값 (`tau / vote_n / debounce_k / fallback_fps`) | `run_summary.json.hyperparameters` | 충분 | 현재 run에서는 후처리 해석이 가능하다. 과거 run은 불연속이 있다. |

## 요약

현재 로그는 `dataset 규모`, `loss recipe`, `후처리값`, `checkpoint 검증` 쪽은 많이 좋아졌다.  
반면 연구적으로 가장 아쉬운 부분은 아래 두 가지다.

1. 라벨 분포가 공식 메타에 없다.
2. 전처리 family가 무엇을 의미하는지 메타만으로는 설명되지 않는다.

즉 지금은 “무슨 설정으로 돌렸는가”는 알 수 있지만, “왜 그 설정이 효과가 있었는가”를 말하려면 아직 추가 정리가 필요하다.
